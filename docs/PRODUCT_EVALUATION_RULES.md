# V2.x 产品评价规则

本文件是项目本地的固定评价规则。任何人修改核心业务后，都必须按这里的规则生成或更新评价报告，避免只凭主观感觉判断版本质量。

## 评价触发条件

必须评价：

- 生产发布前。
- 生产发布后。
- 修改导入、审阅、施工采集、异常工单、导出、权限、存储、数据库、服务器部署后。
- 完成一轮结构性重构后，例如静态 HTML 迁移 Vue、JSON 状态迁移 PostgreSQL。
- 正式业务运行期间每周至少一次。

可简化评价：

- 纯文档修改。
- 注释修改。
- 不影响业务流程的小样式修正。

简化评价也需要在最近一份评价报告的“变更记录”中追加说明。

## 报告位置

评价报告统一保存到：

```text
docs/evaluations/
```

命名规则：

```text
YYYY-MM-DD-product-evaluation.md
YYYY-MM-DD-HHMM-product-evaluation.md
```

同一天多次评价时使用带时间的文件名。

## 执行入口

标准执行：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\run-product-evaluation.ps1 -Change "说明本次修改"
```

只生成报告、不运行自动检查：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\run-product-evaluation.ps1 -SkipChecks -Change "说明本次修改"
```

直接运行 Python：

```powershell
python scripts/generate_product_evaluation.py --run-checks --change "说明本次修改"
```

如需指定状态文件：

```powershell
python scripts/generate_product_evaluation.py --run-checks --state C:\path\local_state.json --users C:\path\users.json
```

## 评分权重

每项 0-100 分，总分按权重计算，权重总和固定为 100。

| 项目 | 权重 | 评价重点 |
| --- | ---: | --- |
| 工程管理适配度 | 8 | 是否贴合现场施工、审阅、异常闭环、终端任务管理 |
| 产品结构 | 7 | 管理员、审阅员、施工员是否各走自己的主流程 |
| 审阅效率 | 8 | 快捷键、切组、归档、补图、异常回退是否足够快 |
| 施工采集体验 | 8 | 手机采集、扫码、拍照、缓存、上传是否适合现场 |
| 离线缓存能力 | 6 | 断网缓存、详情编辑、统一上传、失败重试是否可靠 |
| 数据结构 | 9 | 团队、用户、任务、资料组、照片、事件、工单是否结构化 |
| 代码结构 | 8 | 前后端职责、状态仓储、导入队列、存储抽象是否可维护 |
| 前端结构 | 6 | Vue 组件、响应式、图片预览、状态同步、卡顿控制 |
| 小程序/移动端结构 | 5 | 小程序或移动端采集是否与网页规则同步 |
| 服务器结构 | 6 | Nginx、FastAPI、PostgreSQL、备份、服务守护是否稳定 |
| 图片存储结构 | 6 | OSS/local/url 来源、缩略图、预览图、导出图是否一致 |
| 安全权限 | 6 | 多团队隔离、角色权限、上传访问、生产密钥、后台入口 |
| 运维备份 | 5 | 备份、恢复、健康检查、磁盘风险、发布回滚 |
| 可观测性 | 4 | 日志、导入导出进度、异常统计、卡顿定位能力 |
| 测试覆盖 | 4 | 单元测试、接口回归、移动端手测路径、导出校验 |
| 成本控制 | 4 | 当前预算内是否稳定运行，OSS/服务器成本是否可控 |

## 总分等级

| 总分 | 等级 | 判断 |
| ---: | --- | --- |
| 90-100 | A | 可规模化生产，主要做优化和自动化 |
| 80-89 | B | 可稳定生产，仍需工程化补强 |
| 70-79 | C | 小规模生产可用，关键风险必须排期 |
| 60-69 | D | 高风险可用，只适合短期低压运行 |
| 0-59 | E | 不建议继续上线，先修复核心结构 |

## 85+ 硬门槛

项目总分要达到 85 分以上，必须满足以下工程门槛。门槛未满足时，不允许靠主观调高小项分数冲到 85。

Vue 前端门槛：

- `v2-web` 是生产前端入口。
- 生产页面登记在 `v2-web/src/router/staticPages.ts`。
- 新功能不得继续写入大型静态 HTML。
- 前端结构完成前必须通过：

```powershell
python scripts/verify_vue_migration_gate.py --strict-native
```

去 JSON 化门槛：

- PostgreSQL migration 覆盖 `teams / users / tasks / groups / photos / review_events / audit_events`。
- 照片表必须包含照片级去重、源 URL 哈希、导入批次、软删除字段。
- 重复导入必须以照片级指纹判断，不能只按表号整组跳过。
- `STATE_BACKEND=postgres` 模式下，JSON 不得继续作为事实源。

生产运维门槛：

- 有备份、恢复、健康检查和发布回滚说明。
- 图片存储路径、OSS 引用和本地上传引用可追溯。
- 核心链路测试覆盖迁库、导入去重、审阅、施工上传、异常回退、导出。

扣分上限：

- 如果存在 legacy bridge 页面，前端结构最高按 72 分计。
- 如果仍以 JSON 为主要事实源，数据结构最高按 75 分计。
- 如果照片级去重字段不完整，图片存储结构最高按 78 分计。
- 如果没有备份和恢复流程，运维备份最高按 65 分计。

## 业务红线

评价时必须检查这些规则：

- 总清单是安装地址唯一来源。
- 安装地址不去重，与表号一一对应。
- 展示表号使用总清单原始表号。
- 长条码匹配键：去前 11 位和最后 1 位。
- 总清单短表号匹配键：去前 2 位。
- 导入按照照片级去重，不能只按表号整组跳过。
- 新增照片应补入已有资料组，不重复上传 OSS。
- 审阅分类必须支持快捷键。
- 回退未施工必须清空采集器、模块号和活跃照片，但保留审计。
- 回退异常必须保留资料组并生成异常工单。
- 施工任务由管理员指派，不由施工员自由领取。

## 定期评价流程

每次完成一轮修改后，按以下顺序执行：

1. 运行与本次改动相关的测试或手测。
2. 运行 `scripts\run-product-evaluation.ps1` 生成评价报告。
3. 检查报告中的分数、优势、不足、高风险问题和下一步升级路径。
4. 如果总分低于目标，下轮优先处理扣分最高且不增加预算的问题。

周期要求：

- 正式业务运行期间，每周至少评价一次。
- 生产发布前必须评价一次。
- 生产发布后必须评价一次。
- 涉及数据、导入、审阅、施工采集、异常、导出、权限、OSS、数据库、服务器的改动完成后必须评价一次。

评价结果使用规则：

- 自动检查失败的项目不得通过主观描述加分。
- 存在 legacy bridge 页面时，不得把前端结构按完全工程化评分。
- 仍以 JSON 为事实源时，不得把数据结构按完全工程化评分。
- 升级路径必须可执行，并优先选择不增加预算的方案。

## 升级路径排序原则

后续建议必须按这个顺序排：

1. 先保数据不丢。
2. 再保现场能用。
3. 再保审阅效率。
4. 再做结构工程化。
5. 最后做视觉和体验细节。

任何会增加预算的建议，必须写明成本、收益和是否可以延后。
