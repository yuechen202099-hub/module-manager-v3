# Team Handoff Prompts

## 给小程序工程师团队 Codex

你正在协作 `module-manager-v3` 生产维护仓库。请先从 GitHub 拉取最新生产分支 `production/V3/3.0.72`，基于它创建功能分支 `mp/<短功能名>`，不要直接提交到生产分支。

工作边界：

- 你负责小程序端及其必要的桥接接口适配。
- 当前生产后端已包含 `V3.0.69` 小程序接口基线：微信绑定登录、一键登录、任务列表、资料组列表、单文件上传桥接和活动接口。
- 不要修改正式生产版本号，不要创建生产 tag，不要发布服务器。
- 不要提交 `.env`、密钥、真实数据、上传图片、数据库 dump、构建压缩包。
- 如需改 API、schema、env、权限或数据结构，必须在 PR 描述中写清楚。

交付方式：

1. 从 `production/V3/3.0.72` 切 `mp/<短功能名>`。
2. 小步提交，写清中文变更说明。
3. 完成后提交 PR 到当前生产分支，或提供 patch。
4. PR/patch 必须包含：基线 commit、修改文件清单、功能说明、测试命令和结果、风险与回滚说明。
5. 生产维护 Codex 会统一审阅、分配下一个生产版本号并上线。

## 给项目管理平台工程师团队 Codex

你正在基于 `module-manager-v3` 最新生产版开发项目管理平台能力。请先从 GitHub 拉取最新生产分支 `production/V3/3.0.72`，基于它创建功能分支 `pm-platform/<短功能名>`，不要直接提交到生产分支。

工作边界：

- 平台开发必须持续跟随生产线，不能把开发环境内容反向污染生产。
- 平台功能优先做成可独立合并、可回滚的小增量。
- 不要修改正式生产版本号，不要创建生产 tag，不要发布服务器。
- 不要提交 `.env`、密钥、真实数据、上传图片、数据库 dump、构建压缩包。
- 涉及项目配置、权限、数据库、导入导出或核心业务规则时，必须写迁移说明和回滚说明。

交付方式：

1. 从 `production/V3/3.0.72` 切 `pm-platform/<短功能名>`。
2. 开发期间定期合并或 rebase 最新 `production/V3/<version>`。
3. 完成后提交 PR 到当前生产分支，或提供 patch。
4. PR/patch 必须包含：基线 commit、修改文件清单、功能说明、测试命令和结果、风险与回滚说明。
5. 生产维护 Codex 会统一审阅、分配下一个生产版本号并上线。
