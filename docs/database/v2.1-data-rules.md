# V2.1 Data Rules Report

Scope: local simulation to production database mapping, task status flow, photo-insufficient and supplemental-photo rules, and the single source of truth for installation address.

## Local Simulation To Production Mapping

| Local simulation field | Production table.field | Rule |
| --- | --- | --- |
| `projects[].id` | `projects.id` | Local integer ids are temporary handles only. Production uses UUID. |
| `projects[].name` | `projects.name` | The V2.1 local project name may be used as a seed value, not as a unique key. |
| `tasks[].id` | `tasks.id` | Local integer ids are temporary handles only. Production uses UUID. |
| `tasks[].status` | `tasks.status` | Local `published` and `released` keep the same values. Local `in_review` maps to production `claimed`. |
| `tasks[].claimed_by` | `tasks.claimed_by_id` | Local reviewer name must resolve to a production user id before persistence. |
| `groups[].id` | `material_groups.id` | Local `g-xxxxx` ids are temporary handles only. Production uses UUID. |
| `groups[].task_id` | `material_groups.task_id` and `task_groups.task_id` | Production must persist both the group owner task pointer and task-group membership when publishing. |
| `groups[].meter_match_key` | `material_groups.meter_match_key` | Generated by backend normalization and persisted. Unique per `project_id`. |
| `groups[].meter_no` | `material_groups.display_meter_no` | Must come from matched `total_catalog_rows.original_meter_no` when a total row exists. |
| `groups[].address` | `material_groups.installation_address` | Must come only from matched `total_catalog_rows.installation_address`. |
| `groups[].stage_meter_no` | `stage_catalog_rows.original_barcode` or `raw_data` | Stage data is traceability input, not the display meter source after matching. |
| `groups[].stage_terminal` | `stage_catalog_rows.terminal_no` | Preserved for traceability and task slicing. |
| `groups[].photo_count` | `material_groups.photo_count` | Count only valid uploaded photos after group-level dedupe. |
| `groups[].photos[]` | `photos` | Persist one row per valid photo object with `group_id`, `sha256`, `object_key`, and upload metadata. |
| `scan_unmatched[]` | `exceptions` | Create an exception with category `scan_unmatched` or record in import result details. |
| `stage_unmatched[]` | `exceptions` | Create an exception with category `stage_unmatched`; group status must not be approved. |
| `review_events[]` | `review_records` | Persist immutable review history with previous status, next status, reviewer, notes, and payload. |

## Status Flow

### Task Status

Production task statuses are `draft`, `published`, `claimed`, `released`, `completed`, and `cancelled`.

Allowed V2.1 transitions:

| From | Action | To | Required rule |
| --- | --- | --- | --- |
| `draft` | publish | `published` | Task groups are generated in the same transaction. |
| `published` | claim | `claimed` | Set `claimed_by_id` and `claimed_at`. |
| `released` | claim | `claimed` | Set `claimed_by_id` and `claimed_at`. |
| `claimed` | release by current claimant | `released` | Clear `claimed_by_id` and `claimed_at`; set `released_at`. |
| `claimed` | complete | `completed` | All required groups are approved or explicitly resolved by an authorized exception path. |
| any active status | cancel | `cancelled` | Cancellation requires audit logging and must not delete groups or review records. |

Local simulation uses `in_review` for a claimed task. Production must store this as `claimed`; `in_review` is a presentation/local-only alias for task data.

### Group Status

Production group statuses are `unreviewed`, `in_review`, `incomplete`, `approved`, and `rejected`.

Local aliases:

| Local status | Production status | Notes |
| --- | --- | --- |
| `pending` | `unreviewed` | Matched and has enough photos, awaiting review. |
| `incomplete` | `incomplete` | Matched but valid photo count is below the minimum. |
| `unmatched` | `rejected` plus exception | Stage row has no total-catalog match; it cannot be approved until data is corrected. |
| `approved` | `approved` | Reviewer accepted the group. |
| `exception` | `rejected` plus exception | Local exception marker maps to formal rejected state and an open exception record. |

## Photo-Insufficient And Supplemental-Photo Rules

- The minimum valid photo count for a reviewable group is 4.
- Only photos with `upload_status = uploaded` and a non-empty `object_key` count toward `material_groups.photo_count`.
- Duplicate photos are suppressed per group by `photos(group_id, sha256)`. The same file hash may exist in a different group only when it is intentionally attached there.
- If a matched group has fewer than 4 valid photos, set `material_groups.status = incomplete`, update `photo_count`, and create or keep an open exception with category `photo_insufficient`.
- A group in `incomplete` status cannot be approved.
- Supplemental photo upload appends new `photos` rows to the same `material_groups.id`; it must not create a replacement group and must not change `meter_match_key`.
- After supplemental photos are completed, recalculate `photo_count`. If the count is at least 4 and no other blocking exception remains, move the group from `incomplete` to `unreviewed` for normal review.
- Every supplemental-photo completion must be auditable through `audit_logs`; any review result after补图 must also create a `review_records` entry.

## Total-Catalog Address Single Source Rule

- `total_catalog_rows.installation_address` is the only authoritative source for installation address.
- Stage catalog, scan data, frontend edits, and review comments must not overwrite `material_groups.installation_address`.
- On group creation, copy `total_catalog_rows.installation_address` into `material_groups.installation_address` as a snapshot.
- If duplicate total-catalog rows share a `meter_match_key`, the import/matching policy must select the source row deterministically and record the chosen `total_catalog_row_id`. Address selection must never depend on stage row order or scan row order.
- Catalog tables intentionally do not dedupe or unique-constrain installation address because different meters can share the same address.

## Test Recommendations

- Verify local-to-production status aliasing: task `in_review` maps to `claimed`; group `pending` maps to `unreviewed`; local `exception`/`unmatched` do not map to `approved`.
- Verify the formal database model contains the required production enum values and uniqueness constraints.
- Verify photo insufficiency at 0, 1, 3, and 4 photos, including the transition back to `unreviewed` after supplemental upload.
- Verify duplicate photo hashes are rejected only within the same group.
- Verify `material_groups.installation_address` can be derived only from the matched total-catalog row.
- Verify imported total-catalog rows with duplicate addresses are accepted.

## Risks

- Current local simulation task status `in_review` differs from production `claimed`; API consumers need one canonical production value before formal persistence is enabled.
- Current local simulation marks unmatched groups separately, while production enum has no `unmatched`; this report maps unmatched to `rejected` plus an exception, but product owners should confirm the user-facing label.
- The minimum photo count is documented as 4 because current local simulation uses `< 4` as incomplete. If field policy changes, tests and contracts must change together.
- Duplicate total-catalog meter keys need a deterministic winner rule before production imports can be considered fully specified.
